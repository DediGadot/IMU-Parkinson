## Iter29B Multi-Task RegressorChain — 5-LLM Synthesis

**TL;DR**: The +0.051 5-fold lift is **real within 5-fold** but structurally unlikely to survive LOOCV at +0.025+. The canonical T1 ceiling **remains 0.6550** (iter12 honest). Iter29B is a strong *exploratory* candidate but **not lockboxable** without additional hardening.

---

### 1. Mechanism — What is RegressorChain doing differently?

**Consensus across codex/gemini/grok/kimi**: The lift is **(a) information sharing across item heads**, primarily via conditioning on prior items in the chain — not (b) joint optimization, not (c) effective N×6 sample size increase.

T1 items 9-14 are clinically correlated (r≈0.65-0.82). In the chain:
- **Later heads** see predictions from earlier heads as extra covariates, creating a low-dimensional learned severity proxy.
- This encodes *between-item residual correlation* that independent per-item models (F53) or single-task T1 LGB miss.
- As codex notes: `"RegressorChain is not joint optimization. It fits six scalar LGB models sequentially, so it does not make the effective N equal to 94 × 6 = 564."`

**Critical caveat — Exposure Bias (gemini / codex)**: `sklearn.RegressorChain(cv=None)` appends **true** prior-item residuals during training, but **predicted** residuals at test time. This is not classic leakage, but it causes **teacher-forcing mismatch** that inflates 5-fold stability. The K=500 selector may over-index on these perfect labels.

**Your code analysis confirms**: `run_t1_iter29b_multitask_lgb.py:94` uses default `RegressorChain(base, order="random", random_state=seed)` — `cv=None`.

---

- F56 (nested hybrid k=19): 5-fold Δ=+0.05 → LOOCV collapsed to −0.147 (Ridge meta overfit)

This is the same pattern as prior mixer failures: 5-fold looks promising because fold-level correlations are stable with N≈75 train, but LOOCV (N=1 test) exposes per-fold prediction variance.

### 3. Leakage Paths — Which are real?

**Not leakage (all consultants agree)**:
- ✅ Per-fold K=500 on train residual — fold-local, same as iter5
- ✅ Item residual = item minus train-fold mean — fold-local
- ✅ Random chain order per seed — not data-dependent selection

**Real methodological issues (not classic leakage but structurally inflating)**:
- ⚠️ **Exposure bias / teacher-forcing** (`cv=None`): Training sees true prior items; test sees predicted. This inflates 5-fold stability. Fix: `RegressorChain(..., cv=5)` or manual OOF prior-item covariates inside each outer train.
- ⚠️ **Parallel ensemble / double-counting variance** (gemini): Your Stage 2 predicts `item - train_mean` (not Stage 1 residual), so `final_pred = Stage1 + Stage2` creates an unweighted parallel ensemble. This artificially inflates prediction variance → calibration slope drops to ~0.80 (observed). Fix: Have Stage 2 predict the *actual Stage 1 per-item residual*.
- ⚠️ **Post-hoc variant selection** (grok/kimi): You ran A/B/C (pairwise/multi-task/CCC-direct) in parallel on the same splits and selected B as the winner. **This IS the iter11A pattern** (AGENTS.md:14-15 — multi-layer adaptive selection; bootstrap inflation +0.070). Even if LOOCV runs clean numerically, the *decision to lockbox* was made after seeing 5-fold screen results.

**Required additional null gates**:
- `RegressorChain(cv=5)` rerun
- Independent per-item ablation (same selector/base, no chain covariates)
- Permute prior-item covariates inside train → lift should disappear
- Oracle diagnostic: use true prior items at test to bound max chain benefit
- 100+ scrambled-label permutations (N=94 CCC nulls are noisy; 1-shot is weak)
